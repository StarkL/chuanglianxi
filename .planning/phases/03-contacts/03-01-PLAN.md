<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| WeChat API ŌåÆ Frontend | `encryptedData` + `iv` returned ŌĆö must not be parsed client-side |
| Frontend ŌåÆ Backend | POST /contacts/import-from-phone ŌĆö untrusted body (code, encryptedData, iv) |
| Backend ŌåÆ WeChat API | code2Session call ŌĆö server-to-server with app secret |
| Backend ŌåÆ Database | Contact creation ŌĆö userId isolation required |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Information Disclosure | wechat-crypto.ts | mitigate | Never log sessionKey, encryptedData, or decrypted phone numbers. Use `requireAuth` to protect endpoint. |
| T-03-02 | Tampering | POST /contacts/import-from-phone | mitigate | Fastify body schema validates required fields (code, encryptedData, iv). Decryption failure returns 400 ŌĆö invalid data cannot bypass. |
| T-03-03 | Spoofing | Import endpoint | mitigate | `preHandler: [requireAuth]` ensures only authenticated users can import. userId isolation prevents cross-user data creation. |
| T-03-04 | Repudiation | Session key retrieval | accept | code2Session called per-request with fresh code. If WeChat API is down, error returns gracefully. No audit logging needed for V1. |
</threat_model>
