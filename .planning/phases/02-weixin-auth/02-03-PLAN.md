2. Create frontend/src/utils/auth.ts — token management using uni-app storage:

```typescript
import type { UserInfo } from '../types/auth.js'

const TOKEN_KEY = 'token'
const USER_INFO_KEY = 'userInfo'

export function getToken(): string | null {
  return uni.getStorageSync(TOKEN_KEY) || null
}

export function setToken(token: string): void {
  uni.setStorageSync(TOKEN_KEY, token)
}

export function getUserInfo(): UserInfo | null {
  try {
    const raw = uni.getStorageSync(USER_INFO_KEY)
    return raw ? (JSON.parse(raw) as UserInfo) : null
  } catch {
    return null
  }
}

export function setUserInfo(user: UserInfo): void {
  uni.setStorageSync(USER_INFO_KEY, JSON.stringify(user))
}

export function clearAuth(): void {
  uni.removeStorageSync(TOKEN_KEY)
  uni.removeStorageSync(USER_INFO_KEY)
}

export function isLoggedIn(): boolean {
  return getToken() !== null
}
```

Key points:
- Uses uni.setStorageSync/getStorageSync (not localStorage — this is a mini-program)
- UserInfo stored as JSON string (uni storage only supports primitives)
- clearAuth removes BOTH token and user info
- isLoggedIn checks token presence (simple, no server round-trip)
- Per UI-SPEC: token key is 'token', user info key is 'userInfo'

1. Create frontend/src/utils/request.ts — wrapper around uni.request that automatically adds Authorization header and handles responses:

```typescript
import type { ApiResponse } from '../types/auth.js'
import { getToken, clearAuth } from '../utils/auth.js'

// TODO: Replace with actual backend URL based on environment
const BASE_URL = 'http://localhost:3000'

interface RequestOptions {
  url: string
  method?: 'GET' | 'POST' | 'PUT' | 'DELETE'
  data?: Record<string, unknown>
}

export async function request<T>(options: RequestOptions): Promise<ApiResponse<T>> {
  const token = getToken()

  return new Promise<ApiResponse<T>>((resolve, reject) => {
    uni.request({
      url: `${BASE_URL}${options.url}`,
      method: options.method || 'GET',
      data: options.data,
      header: {
        'Content-Type': 'application/json',
        ...(token ? { Authorization: `Bearer ${token}` } : {}),
      },
      success: (res) => {
        if (res.statusCode === 401) {
          clearAuth()
          uni.redirectTo({ url: '/pages/login/login' })
          reject(new Error('登录已过期'))
          return
        }

        const data = res.data as ApiResponse<T>
        if (data.success) {
          resolve(data)
        } else {
          reject(new Error(data.error || '请求失败'))
        }
      },
      fail: (err) => {
        reject(new Error('网络连接失败'))
      },
    })
  })
}
```

2. Create frontend/src/api/auth.ts — typed auth API calls:

```typescript
import { request } from '../utils/request.js'
import type { LoginRequest, LoginResponse, UserInfo } from '../types/auth.js'

export async function wechatLogin(code: string): Promise<LoginResponse> {
  const res = await request<LoginResponse>({
    url: '/api/auth/wechat-login',
    method: 'POST',
    data: { code },
  })
  return res.data!
}

export async function verifyAuth(): Promise<{ user: UserInfo }> {
  const res = await request<{ user: UserInfo }>({
    url: '/api/auth/verify',
    method: 'GET',
  })
  return res.data!
}

export async function logout(): Promise<void> {
  await request<void>({
    url: '/api/auth/logout',
    method: 'POST',
  })
}
```

Key points:
- BASE_URL uses localhost for development — should be configurable via env var in production
- Automatic 401 handling: clears auth, redirects to login (per UI-SPEC auth guard spec)
- Network errors caught and rejected with Chinese error message
- Typed generic request wrapper returns ApiResponse<T>
- Auth API functions are thin wrappers over request() with typed parameters

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| uni.setStorageSync | Token stored in mini-program local storage — accessible by the mini-program runtime |
| request() wrapper | Attaches token to every request — ensure BASE_URL points to HTTPS in production |
| 401 auto-redirect | Clears local auth state on unauthorized — prevents stale session |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-11 | Information Disclosure | auth.ts (storage) | mitigate | Token stored via uni.setStorageSync (mini-program sandbox); no sensitive data logged or stored beyond token |
| T-02-12 | Spoofing | request.ts BASE_URL | mitigate | BASE_URL must use HTTPS in production; localhost only for development — add env-based config before production deployment |
| T-02-13 | Tampering | request.ts 401 handler | mitigate | 401 response triggers immediate clearAuth + redirect; no retry loop that could leak tokens |
| T-02-14 | Elevation of Privilege | request.ts header injection | accept | uni.request API does not allow arbitrary header manipulation by user input; Content-Type and Authorization are the only headers set |
</threat_model>
