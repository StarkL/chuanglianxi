Create backend/src/middleware/auth.ts — a Fastify preHandler middleware that validates the Authorization header:

```typescript
import type { FastifyRequest, FastifyReply } from 'fastify'
import { verifyToken } from '../lib/auth.js'

declare module 'fastify' {
  interface FastifyRequest {
    user: {
      userId: string
      openId: string
    }
  }
}

export async function requireAuth(request: FastifyRequest, reply: FastifyReply) {
  const authHeader = request.headers.authorization
  if (!authHeader) {
    return reply.status(401).send({
      success: false,
      error: '未提供认证令牌',
      data: null,
    })
  }

  const token = authHeader.replace('Bearer ', '')
  if (token === authHeader) {
    return reply.status(401).send({
      success: false,
      error: '认证格式无效',
      data: null,
    })
  }

  try {
    const payload = await verifyToken(token)
    request.user = {
      userId: payload.sub,
      openId: payload.openId,
    }
  } catch {
    return reply.status(401).send({
      success: false,
      error: '登录已过期',
      data: null,
    })
  }
}
```

Key points:
- Extends FastifyRequest type with `user` property (TypeScript module augmentation)
- Consistent error response format: `{ success: false, error: string, data: null }`
- Error messages in Simplified Chinese (per UI-SPEC copywriting contract)
- Bearer token scheme (standard, not cookie-based)

Create backend/src/routes/auth.ts with the wechat-login endpoint:

```typescript
import type { FastifyInstance } from 'fastify'
import { z } from 'zod'
import { code2Session } from '../lib/wechat.js'
import { generateToken } from '../lib/auth.js'
import { prisma } from '../lib/prisma.js'

const loginSchema = z.object({
  code: z.string().min(1, '登录凭证不能为空'),
})

export async function authRoutes(fastify: FastifyInstance) {
  fastify.post('/auth/wechat-login', {
    schema: {
      body: {
        type: 'object',
        required: ['code'],
        properties: {
          code: { type: 'string' },
        },
      },
      response: {
        200: {
          type: 'object',
          properties: {
            success: { type: 'boolean' },
            data: {
              type: 'object',
              properties: {
                token: { type: 'string' },
                user: {
                  type: 'object',
                  properties: {
                    id: { type: 'string' },
                    openId: { type: 'string' },
                    nickname: { type: ['string', 'null'] },
                    avatar: { type: ['string', 'null'] },
                  },
                },
              },
            },
            error: { type: ['string', 'null'] },
          },
        },
      },
    },
  }, async (request, reply) => {
    const parsed = loginSchema.safeParse(request.body)
    if (!parsed.success) {
      return reply.status(400).send({
        success: false,
        error: parsed.error.errors[0].message,
        data: null,
      })
    }

    const { openId } = await code2Session(parsed.data.code)

    let user = await prisma.user.findUnique({ where: { openId } })

    if (!user) {
      user = await prisma.user.create({
        data: { openId, subscriptionTier: 'free' },
      })
    }

    const token = await generateToken({ sub: user.id, openId: user.openId })

    return {
      success: true,
      data: {
        token,
        user: {
          id: user.id,
          openId: user.openId,
          nickname: user.nickname,
          avatar: user.avatar,
        },
      },
      error: null,
    }
  })
}
```

Key points:
- Zod validation before processing — fail fast with clear message
- Idempotent: existing users found by openId, new users created automatically
- Returns token + user object in consistent ApiResponse envelope
- Does NOT return session_key to client (security)
- Schema defined for Fastify validation + OpenAPI/Swagger docs

Add two more endpoints to backend/src/routes/auth.ts:

**GET /api/auth/verify** — validates token and returns current user:
- Uses requireAuth middleware (preHandler)
- Looks up user from DB by request.user.userId
- Returns user object or 404 if user not found
- Response: `{ success: true, data: { user }, error: null }`

**POST /api/auth/logout** — invalidates session in Redis:
- Uses requireAuth middleware
- Stores the token in a Redis denylist with TTL matching token expiration (7 days)
- For Phase 2: Redis denylist key pattern `auth:blacklist:{token_hash}` (hash token to avoid storing full JWT)
- Response: `{ success: true, data: null, error: null }`
- Future auth checks (in requireAuth) will check Redis denylist — but for Phase 2, logout is client-side only (token cleared from localStorage). Redis blacklist is prepared for Phase 7 (subscription) when server-side revocation becomes critical.

| Authorization header | Untrusted; must verify signature and expiration before trusting user identity |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-06 | Spoofing | /auth/wechat-login | mitigate | Zod schema validates code is non-empty string; code2Session validates with WeChat; no openId accepted without WeChat verification |
| T-02-07 | Replay | JWT token | mitigate | 7-day expiration limits replay window; Redis blacklist enables server-side revocation on logout |
| T-02-08 | Authorization Bypass | requireAuth middleware | mitigate | Every protected route uses requireAuth as preHandler; no route bypasses this; token verified cryptographically via jose |
