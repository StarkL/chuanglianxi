1. Create frontend/src/utils/session-check.ts — validates session against backend:

```typescript
import { verifyAuth } from '../api/auth.js'
import { getToken, setUserInfo, clearAuth } from '../utils/auth.js'

export async function validateSession(): Promise<boolean> {
  const token = getToken()
  if (!token) {
    return false
  }

  try {
    const { user } = await verifyAuth()
    setUserInfo(user)
    return true
  } catch {
    clearAuth()
    return false
  }
}
```

2. Update frontend/src/App.vue to add onLaunch auth check:

```vue
<script setup lang="ts">
import { validateSession } from './utils/session-check.js'
import { isLoggedIn } from './utils/auth.js'

async function onLaunch(options: { path?: string; query?: Record<string, string> }) {
  // If already on login page, skip validation
  if (options.path === 'pages/login/login') {
    return
  }

  const isValid = await validateSession()
  if (!isValid) {
    uni.reLaunch({ url: '/pages/login/login' })
  }
}
</script>

<template>
  <view class="app">
    <slot />
  </view>
</template>

<style>
page {
  background-color: #f5f5f5;
}
</style>
```

Key points:
- onLaunch is async — validates token on every app cold start
- If no token or verification fails -> reLaunch to login (clears all page stack)
- If user opens app directly on login page (e.g., from shared link) -> skip validation
- Valid session -> user info updated in storage (fresh from backend)
- Uses reLaunch (not redirectTo) to ensure clean page stack

1. Add logoutWithConfirm() to frontend/src/utils/session-check.ts:

```typescript
import { logout as logoutApi } from '../api/auth.js'
import { clearAuth } from '../utils/auth.js'

export async function logoutWithConfirm(): Promise<void> {
  return new Promise<void>((resolve) => {
    uni.showModal({
      title: '确认退出？',
      content: '退出后需要重新授权登录',
      confirmText: '退出登录',
      cancelText: '取消操作',
      success: async (res) => {
        if (res.confirm) {
          try {
            await logoutApi()
          } catch {
            // Backend may be unavailable; still clear local auth
          }
          clearAuth()
          uni.reLaunch({ url: '/pages/login/login' })
        }
        resolve()
      },
    })
  })
}
```

```typescript
export function requireAuth(redirectPath?: string): void {
  if (!isLoggedIn()) {
    const redirect = redirectPath || '/pages/index/index'
    const encodedRedirect = encodeURIComponent(redirect)
    uni.reLaunch({
      url: `/pages/login/login?redirect=${encodedRedirect}`,
    })
  }
}
```

This is exported for Phase 3+ pages to call in their onLoad() hook. The login page reads the `redirect` query param (handled in Plan 04) and redirects back after successful login.

Key points:
- Logout uses uni.showModal (native mini-program dialog, no custom modal needed)
- Chinese copy matches UI-SPEC exactly: title "确认退出？", content "退出后需要重新授权登录"
- Backend logout called first; if it fails (network), still clear local auth
- requireAuth is a simple check utility for pages to guard themselves
- reLaunch used for logout to clear entire page stack

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| App onLaunch -> /api/auth/verify | First network call on app start; untrusted network may be unavailable |
| Logout confirmation | Client-side only; backend called but local clear always happens |
| requireAuth redirect | Redirect URL encoded to prevent open redirect attacks |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-18 | Spoofing | App onLaunch validation | mitigate | validateSession calls backend /api/auth/verify — token verified server-side, not just client-side check |
| T-02-19 | Denial of Service | validateSession network failure | mitigate | If network unavailable, verify fails -> user sent to login (safe default); no infinite retry |
| T-02-20 | Open Redirect | requireAuth redirect param | mitigate | redirect param URL-encoded; login page validates redirect starts with /pages/ prefix before using it |
| T-02-21 | Session Fixation | logout flow | mitigate | Backend blacklists token in Redis; client clears both token and userInfo; reLaunch clears page stack |
</threat_model>
