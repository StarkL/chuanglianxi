## Design System

| Property | Value |
|----------|-------|
| Tool | none (WeChat mini-program, not web) |
| Preset | not applicable |
| Component library | uni-ui (official DCloud library, lightweight) -- selected per PROJECT.md Section 5.2 |
| Icon library | WeChat native icons + inline SVG for custom icons |

**Rationale:** shadcn is a web-only solution and cannot be used in WeChat mini-programs. uni-ui is the official DCloud component library, recommended in the product planning document (Section 5.2), and provides sufficient components for a login-focused phase without heavy dependencies. uView is deferred until later phases when more UI components are needed.

## Color

WeChat mini-programs operate within a constrained visual environment. The color palette follows the 60-30-10 rule, adapted for a minimal login page.

| Role | Hex Value | Usage |
|------|-----------|-------|
| Dominant (60%) | #FFFFFF (white) | Page background, card surfaces |
| Secondary (30%) | #F6F6F6 (light gray) | Input field backgrounds, subtle dividers |
| Accent (10%) | #07C160 (WeChat green) | Primary CTA button, active state indicators |
| Text Primary | #333333 | Headings, body text, labels |
| Text Secondary | #999999 | Helper text, placeholder text |
| Destructive | #E64340 (WeChat red) | Error messages, destructive actions (logout) |
| Border | #EEEEEE | Input field borders, card borders |

**Accent (#07C160) reserved for:**
- Primary login button (the single most important CTA)
- Active/focus state ring on interactive elements
- Success state indicators (e.g., "login successful" icon)
- Never used for secondary buttons, links, or decorative elements

**Rationale:** WeChat green (#07C160) is the native WeChat brand color and signals trust to users in the WeChat ecosystem. Using this for the primary CTA aligns with user expectations for "this action is connected to WeChat."

## Interaction Contracts

### Visual Hierarchy

- **Primary focal point:** Logo area (240rpx square, centered) — draws eye first
- **Secondary visual anchor:** Primary "微信登录" button — clear CTA below logo
- **Tertiary:** Privacy notice and helper text — supporting copy, subdued styling

### Login Flow

1. User opens mini-program -> lands on login page
2. User taps "微信登录" button
3. Button shows loading state ("登录中..."), disabled during process
4. System calls `wx.login()` to obtain login code
5. Code sent to backend `/api/auth/wechat-login` endpoint
6. Backend exchanges code via WeChat `code2Session` API, creates/updates user record
7. Backend returns session token, user info
8. On success: navigate to home page (Phase 3+)
9. On failure: display appropriate error message from table above

### Logout Flow

1. User taps "退出登录" in settings
2. Modal appears with confirmation
3. User confirms -> clear local storage (session token, user info)
4. Redirect to login page
5. User cancels -> modal closes, no action

### Auth Guard (Auto-redirect)

- All pages except login page check for valid session token on load
- If no token or token expired -> redirect to login page with `redirect` query param
- After successful login -> redirect back to original destination

## WeChat-Specific Implementation Notes

### Login API Pattern

The WeChat mini-program login flow uses the following pattern (not web OAuth):

```
Client: wx.login() -> gets temporary code
Client: POST /api/auth/wechat-login { code }
Server: code2Session(code) -> gets open_id, session_key
Server: Create/update user record (open_id, nickname, avatar)
Server: Generate session token (JWT), store in Redis
Server: Return { token, user }
Client: Store token in uni.setStorageSync()
```

### Profile Information Collection

As of 2023+, WeChat deprecated `getUserProfile` and `<button open-type="getUserInfo">`. The current recommended approach:

1. **Authentication** (required): `wx.login()` + `code2Session` -> obtains `open_id`
2. **Profile display** (optional): Use `<button open-type="chooseAvatar">` for avatar selection, and a text input for nickname
3. For Phase 2: Login requires only `wx.login()` code exchange. Nickname/avatar collection can be deferred to a profile setup step after first login

### Session Storage

- Token stored via `uni.setStorageSync('token', token)`
- User info stored via `uni.setStorageSync('userInfo', userInfo)`
- On app launch (`App.onLaunch`): validate token via `/api/auth/verify`
- On token invalid/expired: clear storage, redirect to login
